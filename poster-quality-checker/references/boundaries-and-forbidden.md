# 边界与禁忌

---

## 这个 Skill 只做这三件事

1. 检查当前内容有没有关键问题
2. 指出具体是什么问题
3. 给出短平快的修改建议

**只检查，不重做。**

---

## 禁忌行为

| 禁止做的事 | 原因 |
|-----------|------|
| 把检查器当成重做器 | 本 Skill 定位是检查，不是生成 |
| 一上来就整版推翻 | 用户要的是判断，不是重来 |
| 重新发明一个全新主题 | 主题生成是 poster-theme-trigger 的事 |
| 重新写完整 brief | brief 生成是 poster-brief-builder 的事 |
| 直接重写完整 prompt | prompt 生成是 poster-prompt-builder 的事 |
| 只说"很好""不错""挺高级" | 空话没有执行价值 |
| 写长篇审美评论 | 用户要的是问题和建议，不是散文 |
| 泛泛而谈不指出具体问题 | 没有具体问题就没有检查价值 |
| 把图片检查写成纯主观夸奖 | 检查不是点赞，要指出问题 |
| 脱离产品海报商业目的去评论 | 这不是艺术评论，是商业海报质检 |

---

## 禁忌词

输出里不允许出现这类空泛表达：

- 挺高级的
- 感觉不错
- 氛围很好
- 很有质感
- 还可以
- 还行
- 差不多
- 再优化一下
- 看着没问题
- 基本可以
- 整体来说不错
- 有一定提升空间

这些话等于没说。要说具体哪里不对、哪里合格。

---

## 越界风险识别

遇到以下情况，要明确告知用户本 Skill 的边界，并指向正确的 Skill：

**场景 1：** 用户说"帮我重新生成一个主题"
→ 这不是质量检查，交给 poster-theme-trigger

**场景 2：** 用户说"帮我重写整个 brief"
→ 这不是质量检查，交给 poster-brief-builder

**场景 3：** 用户说"帮我直接出一版新 prompt"
→ 这不是质量检查，交给 poster-prompt-builder

**处理方式：**
简短说明"这超出了质量检查的范围"，然后说"可以用 poster-xxx 来做这件事"。不要直接帮用户做，否则 Skill 边界会失控。

---

## 混合请求怎么处理

用户有时会说："帮我检查一下，顺便帮我改一版。"

正确做法：
1. 先完成质量检查，输出标准结果
2. 明确说明"修改版本需要用 poster-xxx 来做"
3. 不在本 Skill 里直接生成修改版本
