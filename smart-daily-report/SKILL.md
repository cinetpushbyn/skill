# 智能复盘日报 (smart-daily-report)

## 技能定位

本技能的核心使命是：将凌乱的会话记录转译为专业、简洁、符合职场规范的日报文本。
不是总结聊天，而是**提炼工作**。最终输出必须像一位严谨的职场人亲手写就，
而非由 AI 拼凑的对话摘要。


---

## 执行流水线（Execution Pipeline）

> **执行准则**：以下六个步骤必须**严格按顺序静默执行**，任何步骤不得跳过或合并。
> "静默执行"意味着中间过程不向用户展示，只在 Step 6 输出最终结果。

---

### Step 1 · 全局上下文嗅探与复盘

**目标**：从当前会话中精准识别今天实际做了哪些工作。

执行要点：
- 遍历**当前会话的全部历史消息**（包括用户输入和助手回复）
- 识别维度包括但不限于：
  - 讨论过的任务、方案、需求
  - 测试过的代码或调试的 Bug
  - 修改过的设计、文案或方案
  - 调优过的提示词或参数
- 按**时间顺序**梳理工作线索，识别哪些已完成、哪些推进中、哪些搁置

> **注意**：如会话内容涉及多个独立项目，应分别识别，不得混淆。

---

### Step 2 · 物理时间锚定

**目标**：获取真实日期，严禁凭模型记忆或推测填写日期。

执行方式（隐式后台执行，不向用户展示命令）：

```bash
# 获取今日日期（格式：YYYY年MM月DD日）
date +"%Y年%m月%d日"

# 获取明日日期（格式：YYYY年MM月DD日）
date -d "+1 day" +"%Y年%m月%d日"
```

- 将上述命令的**真实输出**作为日报中的日期字段
- 若执行失败，使用系统上下文中已提供的 `currentDate` 变量，并据此推算明日
- **绝对禁止**大模型根据训练数据幻觉编造任何日期

---

### Step 3 · 工作量解构（过程 + 结果双轨并行）

**目标**：将原始���话信息降噪并转译为专业工作项描述。

双轨并行原则：
| 轨道 | 说明 | 示例 |
|------|------|------|
| **过程轨** | 反映迭代次数、评审轮次、方案比选等工作过程 | "进行3轮方案评估与调优" |
| **结果轨** | 反映最终交付物或阶段性里程碑 | "最终完成8601型号包装定稿" |

转译规则：
- 将口语化、随意化的表达**强制替换**为职场专业动词（详见「语言规范」一节）
- 若某项工作只有过程无明确结果，标注"（推进中）"
- 若某项工作有明确结果，直接呈现结果，过程可简述

工作项书写格式：
```
[职场动词]+[工作对象]+[过程描述（可选）]+[结果状态]
```

示例：
- ✅ `进行产品包装文案的3轮评估与修改，最终完成8601型号包装定稿`
- ✅ `探讨并初步确认DT560C的外观设计方向`
- ❌ `搞了下包装文案，弄完了`

---

### Step 4 · 常识顺延推演引擎

**目标**：基于今日已完成或推进中的任务，推演出合理的明日计划。

推演逻辑（常见场景示例）：

| 今日工作类型 | 明日推演方向 |
|--------------|--------------|
| 文案/方案定稿 | 提交评审 / 对接下游执行（如排版、打样）|
| 外观设计初探 | 输出初版草图或渲染方案 / 进一步细化视觉细节 |
| Bug 发现与定位 | 修复问题并进行回归测试 |
| 功能开发完成 | 提测 / 撰写测试用例 / 完善文档 |
| 需求评审讨论 | 输出正式需求文档 / 原型设计 |
| 数据分析探索 | 输出分析报告 / 对接业务方 |
| 提示词调优 | 扩大测试集验证 / 部署上线 |

推演准则：
- 推演结果必须**逻辑连贯**，符合对应行业/领域的正常工作流
- 一般推演 1~3 个明日计划项，数量与今日工作量匹配
- 明日计划用词同样遵循职场化规范，不得出现主观语气词

---

### Step 5 · 组装与合规性终审

**目标**：严格按排版规范拼装日报，并在输出前完成最后一次扫描。

执行顺序：
1. 读取 `references/layout-rules.md` 中的排版模板
2. 将 Step 2 获取的日期、Step 3 提炼的工作项、Step 4 推演的计划项填入模板
3. 执行**合规性扫描**，逐条检查以下禁止项：

合规性检查清单：
- [ ] 输出开头无任何问候语（如"你好"、"好的"）
- [ ] 输出结尾无废话（如"以上是您的报告"、"祝工作顺利"、"就这些"）
- [ ] 不含任何解释性对话（如"根据您今天的会话记录..."）
- [ ] 不含任何��观推测语气（如"我推测明天应该..."）
- [ ] 禁忌词表中的词汇均已替换完毕
- [ ] 日期字段使用的是真实获取的日期（非幻觉）

> 如发现任何违规项，在输出前静默修正，不向用户声明修改过程。

---

### Step 6 · 最终纯净输出

**目标**：一次性输出纯净的日报文本，无前缀，无后缀，直接结束响应。

输出规范：
- **第一个字符**即为日报正文的第一个字
- **最后一个字符**即为日报正文的最后一个字
- 不附加任何解释、备注或结尾话术
- 格式严格遵循 `references/layout-rules.md` 中的模板定义


---

## 语言规范：禁忌词与职场化替换

### 绝对禁用词

以下词汇在任何工作项或计划项中**严禁出现**：

```
搞了下、弄了下、随便弄弄、弄完、大概、好像、
你好、以上是报告、祝好、就这些、
我推测明天应该做
```

### 强制替换映射表

| 禁用表达 | 强制替换为 |
|----------|------------|
| 搞 / 弄 / 试了试 | 推进 / 完成 / 优化 / 测试 / 调优 |
| 画图 / P图 | 视觉设计 / 修图 / 排版输出 |
| 改了改 | 迭代修订 / 优化调整 |
| 看了下 / 看了看 | 评估 / 审阅 / 复核 |
| 聊了下 / 讨论了下 | 沟通确认 / 评审 / 研讨 |
| 写了点 | 撰写 / 输出 / 完成初稿 |
| 发现问题 | 识别并定位问题 |
| 没弄完 | 持续推进中 / 阶段性完成 |

## 示例参考（Good Case）

> 此示例展示了"过程 + 结果"双轨记录与"常识顺延推演"的正确写法。
> Agent 执行时应内化此逻辑，而非机械套用示例内容。

### 场景设定

用户今天与 AI 进行了以下工作：
1. 反复修改了 3 次产品包装文案，最终定稿了 8601 型号的包装
2. 讨论了 DT560C 外观设计的一个初步想法

用户输入：`整理日报`

### 期望输出（严格遵照此格式）

```
2026年3月30日工作总结（工作汇总及进度）：
1.进行产品包装文案的3轮评估与修改，最终完成8601型号包装定稿
2.探讨并初步确认DT560C的外观设计方向

2026年3月31日工作计划(明日工作安排）：
1.推进8601型号包装的视觉排版与打样对接
2.输出DT560C外观设计的初版草图或渲染方案
```

### 解析说明

| 要素 | 来源 | 处理方式 |
|------|------|----------|
| `3轮评估与修改` | 会话中的迭代次数 | 过程轨：保留迭代信息 |
| `完成8601型号包装定稿` | 最终结果 | 结果轨：明确交付物 |
| `初步确认DT560C外观设计方向` | 讨论结论 | 结果轨：阶段性里程碑 |
| `视觉排版与打样对��` | 常识推演 | 定稿后的自然下游任务 |
| `初版草图或渲染方案` | 常识推演 | 初探后的自然延续 |
